# Serializer Specification

The key words MUST, MUST NOT, SHOULD, SHOULD NOT, and MAY are normative.

A Serializer projects a domain object into the JSON shape returned to clients.
Presentation only — no queries, no business rules.

## 1. Definition & Location

1.1. A Serializer MUST live under `app/serializers/{version}/`.
     Example: `app/serializers/v1/user_serializer.rb`.

1.2. The class MUST be namespaced by version and named `{Resource}Serializer`
     (singular resource).
     Example: `class V1::UserSerializer < MainSerializer`.

1.3. A Serializer MUST subclass `MainSerializer`, which is
     `MainSerializer < ActiveModel::Serializer`.

## 2. Responsibilities

2.1. A Serializer MUST declare its output via `attributes` and association
     macros only:
```ruby
class V1::UserSerializer < MainSerializer
  attributes :id, :email, :name, :role, :created_at

  has_many :posts, serializer: V1::PostSerializer
  belongs_to :organization, serializer: V1::OrganizationSerializer
end
```

2.2. Associations MUST name the nested serializer explicitly via `serializer:`
     (or `each_serializer:` where applicable).

2.3. A Serializer MUST NOT execute queries. Anything that triggers a DB call
     (counts, lookups, aggregates) MUST be prepared by the Operation and exposed
     on the object/model instead.

2.4. A Serializer MUST NOT contain business rules or non-trivial branching.

2.5. A Serializer MUST NOT mutate state or cause side effects.

## 3. Computed Attributes

3.1. Simple presentation-only derived values MAY be defined as methods:
```ruby
attributes :id, :full_name

def full_name
  "#{object.first_name} #{object.last_name}"
end
```

3.2. A method attribute MUST be pure formatting over already-loaded data. It MUST
     NOT query, decide policy, or load associations lazily (avoids N+1).

3.3. Conditional attribute inclusion SHOULD use Active Model Serializers' own
     mechanisms; complex conditionals are a smell — push the decision upstream.

## 4. Collections vs Resources

4.1. For a single resource the controller passes `serializer:`; for a collection
     it passes the same class via `each_serializer:` (handled by
     `render_collection(serializer:)`). The same Serializer class serves both.

4.2. Pagination metadata is NOT the Serializer's concern. It is built by the
     controller's `pagination_meta` and passed as `meta:`.

## 5. Errors

5.1. Error responses are shaped by `Errorable`, not by resource Serializers. A
     resource Serializer MUST NOT format error payloads. The dedicated
     `ErrorSerializer` handles `render_error`.

## 6. Output Surface

6.1. A Serializer MUST expose only intended public fields. Do not blanket-expose
     every model column; list attributes explicitly.

6.2. Sensitive fields (password digests, tokens, internal flags) MUST NOT be
     declared.

## 7. Generation

Serializers are not auto-generated by the resource generators in the default
flow; create them under `app/serializers/{version}/` following §1–§2 and
reference them from the controller's render calls.
